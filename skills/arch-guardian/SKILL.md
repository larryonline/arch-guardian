# Arch Guardian — 架构守护工具

## 概述

这是一个通用架构审查框架。它不内置特定的架构规范，而是引导你基于项目自身的架构文档和约定来执行审查。

核心理念：架构是活的，需要持续守护。每次变更都是架构劣化的潜在入口，每次审查都是修正航向的机会。

## 审查流程

### 第一步：确定审查模式和深度

**审查模式** — 根据上下文判断：

- **设计审查（前置）** — 用户准备引入新功能/模块/系统，尚未实现。审查设计方案对整体架构的影响，输出风险和隐患。此模式下组件层审查自动跳过（代码还不存在）。
- **实现审查（后置）** — 代码已经写完，审查已有实现是否符合架构规范。三层全部执行。

如果无法判断，询问用户。

**审查深度** — 根据变更规模自动调整：

- **聚焦审查** — 变更范围小（单个文件、单个类、局部修改）。只审查直接相关的审查项，跳过明显无关的项。报告中标注"[跳过：与本次变更无关]"。
- **标准审查** — 变更涉及多个模块或引入新模块。三层全部执行。
- **全面审查** — 用户明确要求全项目扫描，或项目处于重大重构节点。三层全部执行，且对每个审查项深入分析。

用户可以显式指定深度，否则根据变更范围自动判断。

### 第二步：收集审查依据

架构审查需要有据可依。审查依据的收集分为两种情况：

#### 首次使用：项目知识发现

如果 `.claude/arch-guardian/` 目录不存在（即该项目首次使用本技能），执行一次完整的项目知识发现：

**第一轮：自动扫描项目**

扫描项目中可能包含架构指导信息的文档，关注以下类型：

| 类型 | 常见位置 |
|------|----------|
| 架构原则/分层规则 | `CLAUDE.md`、`AGENTS.md`、`docs/architecture/`、`docs/design/`、`README.md` 中的架构章节 |
| 代码约定/代码品味 | `.claude/`、`.editorconfig`、`CONTRIBUTING.md`、`STYLE_GUIDE.md`、linter 配置文件 |
| 技术栈与最佳实践 | `package.json`、`Cargo.toml`、`go.mod`、`project.godot`、框架配置文件等（用于识别技术栈） |
| 架构决策记录（ADR） | `ADR/`、`docs/decisions/`、`docs/adr/` |
| 历史审查报告 | `.claude/arch-guardian/` |

扫描完成后，向用户报告发现结果：找到了哪些文档、识别出的技术栈、缺少哪些类型的文档。

**第二轮：交互补充**

对于扫描中未覆盖的类型，询问用户：
- 是否有额外的架构文档或约定存放在项目外部（如团队 wiki、Confluence 等）
- 项目是否有未成文但团队共识的架构原则或设计偏好
- 对于识别出的技术栈，是否有特定的使用约定或反模式需要关注

**第三轮：主动获取（需用户授权）**

如果经过前两轮仍然缺少关键的架构指导信息（特别是技术栈的最佳实践和惯用模式），向用户建议：

> "项目使用了 [技术栈名称]，但没有找到相关的架构最佳实践文档。是否需要我搜索该技术栈的官方架构指南和社区最佳实践，作为审查参考？"

获得用户同意后，使用 WebSearch 或其他可用的搜索工具获取：
- 该技术栈的官方架构建议和推荐模式
- 社区公认的最佳实践和常见反模式
- 框架/引擎特有的架构约束

将获取的结果摘要呈现给用户确认后，纳入审查依据。

**知识发现结果持久化**

完成知识发现后，将收集到的审查依据索引保存到 `.claude/arch-guardian/guardian-knowledge.md`，内容包括：
- 项目识别出的技术栈
- 找到的参考文档路径及其类型
- 用户补充的未成文约定
- 从外部获取的最佳实践摘要（附来源）

后续审查直接读取此文件，无需重复发现。当项目技术栈或架构文档有重大变化时，用户可以要求重新执行知识发现。

#### 后续使用：读取已有依据

如果 `.claude/arch-guardian/guardian-knowledge.md` 已存在，直接读取作为审查依据。同时快速检查：
- 引用的参考文档是否仍然存在
- 是否有新增的架构相关文档未被索引
- 如果发现变化，提醒用户是否需要更新 `guardian-knowledge.md`

### 第三步：确定审查范围

按优先级确定范围：

1. **用户指定** — 用户明确说了要审查什么，就审查什么
2. **变更聚焦** — 如果有明确的变更上下文（正在讨论的需求、刚完成的代码），以变更为切入点，审查其对整体架构的影响
3. **全项目扫描** — 无任何上下文时，扫描整个项目的目录结构、模块组织、依赖关系

无论哪种范围，都需要先对项目全局有基本了解（目录结构、主要模块、入口点），再聚焦到具体审查目标。

### 第四步：执行审查

对每个审查项，给出以下判定之一：
- **通过** — 符合规范，无需修改
- **警告** — 不违规但存在隐患，建议关注
- **不通过** — 违反规范，需要修改
- **跳过** — 与本次审查范围无关（仅聚焦审查模式下使用）

## 架构层审查

审查整体架构的健康度。

### A1. 分层清晰度
- 项目是否有明确的分层（参照项目架构文档中定义的层级划分）
- 各层职责是否清晰，有无职责混淆
- 层与层之间的依赖方向是否合理（避免反向依赖、循环依赖）

### A2. 目录结构
- 目录组织是否能让新人快速理解项目结构
- 有无"垃圾桶"目录（如 `utils/`、`misc/`、`managers/`、`temp/`）
- 文件放置位置是否符合其职责归属

### A3. 全局依赖
- 全局状态和全局可访问对象的数量和职责是否合理
- 有无"上帝对象"承担了过多职责
- 跨模块通信方式是否统一且可追踪

### A4. 变更影响（设计审查重点关注此项）
- 新增的系统/模块是否在已有分层中有明确归属
- 是否引入了新的架构概念或模式（需要额外审慎）
- 是否打破了已有的架构约定

---

## 模块层审查

审查各模块的内聚性和模块间的耦合度。

### M1. 模块边界
- 每个模块是否有明确的职责范围
- 模块的公开接口是否最小化
- 模块内部实现是否对外隐藏

### M2. 模块间耦合
- 模块之间是否通过明确的接口或约定的通信机制交互
- 有无模块直接操作另一个模块的内部状态
- 修改一个模块是否会波及到不相关的模块

### M3. 可替换性
- 模块是否可以独立理解和测试
- 是否可以替换某个模块的实现而不影响其他部分

### M4. 数据流向
- 数据在模块间的流动路径是否清晰
- 有无数据在多个模块间被不必要地传递或复制
- 配置/数据是否与行为适当分离

---

## 组件层审查

审查具体的代码单元（文件、类、函数）。设计审查模式下跳过此层。

### C1. 单一职责
- 每个文件/类是否只做一件事
- 有无"上帝对象"承担了过多不相关的职责
- 函数长度和复杂度是否合理

### C2. 命名和可读性
- 命名是否能表达语义（而非 `Manager`、`Helper`、`Util` 等模糊名字）
- 代码组织结构是否能直观反映职责

### C3. 依赖方向
- 组件是否只依赖它应该知道的东西
- 有无越级引用（如底层组件直接操作上层对象）
- 依赖注入是否优先于硬编码引用

### C4. 可扩展性
- 新增同类对象的成本是否足够低
- 是否需要大量复制已有代码才能扩展
- 数据驱动的部分是否已适当外置

## 审查报告持久化

审查报告保存到项目中，作为架构演进的记录。

### 存储位置

- **默认目录**: `.claude/arch-guardian/`
- 用户指定了其他目录则使用用户指定的
- 首次使用时自动创建目录

### 文件命名

```
.claude/arch-guardian/
  YYYY-MM-DD-<主题摘要>.md
```

例如：
- `2026-03-18-新增存档系统.md`
- `2026-03-20-全项目架构扫描.md`

### 历史关联

- 新报告应检查是否有相关的历史报告，如有则在报告开头引用，说明架构演进轨迹
- 如果之前的"不通过"项已修复，在旧报告末尾追加更新记录：

```
## 更新记录
- **YYYY-MM-DD**: [更新说明，如"A2 目录结构问题已修复"]
```

## 架构文档资产化

架构知识不应只存在于代码中或审查者的脑中，应当沉淀为可追溯的文档资产。

### 触发时机

审查过程中如果发现以下情况，主动提醒用户：

1. **项目没有架构文档** — 建议用户创建基础架构文档，至少包含：分层定义、目录结构约定、模块职责划分、跨模块通信方式
2. **架构文档与代码不一致** — 代码实际结构已偏离文档描述，提醒用户更新文档
3. **审查中产生了新的架构决策** — 建议用户记录为 ADR（Architecture Decision Record）

### 架构文档同步检查

每次审查时，如果项目有架构文档，额外执行一次同步检查：

- 文档中描述的分层/模块是否与代码实际结构一致
- 文档中的约定是否仍在被遵守
- 是否有新增的模块/系统未被文档覆盖

同步检查结果附在审查报告末尾：

```
## 文档同步状态
- **架构文档**: [路径] — 同步 / 需更新（说明偏差）
- **ADR**: [是否存在] — 最新记录日期
```

## 审查原则

1. **有据可依** — 每个判定都要能追溯到参考文档中的具体条款。如果是基于通用原则，也要明确说明是哪条原则。
2. **就事论事** — 只审查架构层面的问题，不纠结代码风格细节（那是 code review 的事）。组件层的审查也聚焦在架构意义上的问题（职责划分、依赖方向），而非语法偏好。
3. **务实判断** — 不追求理论完美。一个"足够好"的架构优于一个"理论完美但过度设计"的架构。如果某处不够理想但当前规模下完全可控，给"警告"而非"不通过"。
4. **考虑阶段** — 原型阶段和生产阶段的标准不同。对早期项目宽容一些，对成熟项目严格一些。
5. **建议可行** — 给出的建议必须是用户当前就能采取行动的，而不是"以后重写整个项目"。
