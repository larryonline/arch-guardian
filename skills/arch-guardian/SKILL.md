# Arch Guardian — 架构守护工具

## 概述

这是一个通用架构审查框架。它不内置特定的架构规范，而是引导你基于项目自身的架构文档和约定来执行审查。

核心理念：架构是活的，需要持续守护。每次变更都是架构劣化的潜在入口，每次审查都是修正航向的机会。

## 审查流程

### 第一步：确定审查模式和深度

**审查模式** — 根据上下文判断：

- **设计审查（前置）** — 用户准备引入新功能/模块/系统，尚未实现。审查设计方案对整体架构的影响，输出风险和隐患。此模式下组件层审查自动跳过（代码还不存在）。
- **实现审查（后置）** — 代码已经写完，审查已有实现是否符合架构规范。三层全部执行。

如果无法判断，询问用户。

**审查深度** — 根据变更规模自动调整：

- **聚焦审查** — 变更范围小（单个文件、单个类、局部修改）。只审查直接相关的审查项，跳过明显无关的项。报告中标注"[跳过：与本次变更无关]"。
- **标准审查** — 变更涉及多个模块或引入新模块。三层全部执行。
- **全面审查** — 用户明确要求全项目扫描，或项目处于重大重构节点。三层全部执行，且对每个审查项深入分析。

用户可以显式指定深度，否则根据变更范围自动判断。

### 第二步：检查速检 Hook 配置

在执行任何审查之前，先检查变更影响速检的 hook 是否已配置：

1. 读取项目的 `.claude/settings.json`（如不存在则视为未配置）
2. 检查 `hooks.PreToolUse` 中是否有指向 `pre-commit-hook.sh` 的配置项
3. 如果未配置，提示用户：

   > "检测到项目尚未配置变更影响速检 hook。此 hook 会在 git commit 前自动执行轻量级架构检查，帮助及时发现架构劣化。是否现在配置？"

4. 用户同意后，自动完成配置：
   - 确定本技能的 `scripts/pre-commit-hook.sh` 的绝对路径
   - 确定 hook 配置的写入位置：与本技能安装 scope 一致。如果技能安装在用户全局（`~/.claude/`），写入 `~/.claude/settings.json`；如果安装在项目级别（`.claude/`），写入 `.claude/settings.json`。如文件已存在则合并到现有配置中，不覆盖其他 hook

如果用户拒绝，跳过此步骤，继续执行审查。速检 hook 不影响完整审查流程。

### 第三步：收集审查依据

架构审查需要有据可依。审查依据的收集分为两种情况：

#### 首次使用：项目知识发现

如果 `.arch-guardian/` 目录不存在（即该项目首次使用本技能），执行一次完整的项目知识发现：

**第一轮：自动扫描项目**

扫描项目中可能包含架构指导信息的文档，关注以下类型：

| 类型 | 常见位置 |
|------|----------|
| 架构原则/分层规则 | `CLAUDE.md`、`AGENTS.md`、`docs/architecture/`、`docs/design/`、`README.md` 中的架构章节 |
| 代码约定/代码品味 | `.claude/`、`.editorconfig`、`CONTRIBUTING.md`、`STYLE_GUIDE.md`、linter 配置文件 |
| 技术栈与最佳实践 | `package.json`、`Cargo.toml`、`go.mod`、`project.godot`、框架配置文件等（用于识别技术栈） |
| 架构决策记录（ADR） | `ADR/`、`docs/decisions/`、`docs/adr/` |
| 历史审查报告 | `.arch-guardian/reports/` |

扫描完成后，向用户报告发现结果：找到了哪些文档、识别出的技术栈、缺少哪些类型的文档。

**第二轮：交互补充**

对于扫描中未覆盖的类型，询问用户：
- 是否有额外的架构文档或约定存放在项目外部（如团队 wiki、Confluence 等）
- 项目是否有未成文但团队共识的架构原则或设计偏好
- 对于识别出的技术栈，是否有特定的使用约定或反模式需要关注

**第三轮：主动获取（需用户授权）**

如果经过前两轮仍然缺少关键的架构指导信息（特别是技术栈的最佳实践和惯用模式），向用户建议：

> "项目使用了 [技术栈名称]，但没有找到相关的架构最佳实践文档。是否需要我搜索该技术栈的官方架构指南和社区最佳实践，作为审查参考？"

获得用户同意后，使用 WebSearch 或其他可用的搜索工具获取：
- 该技术栈的官方架构建议和推荐模式
- 社区公认的最佳实践和常见反模式
- 框架/引擎特有的架构约束

将获取的结果摘要呈现给用户确认后，纳入审查依据。

**知识发现结果持久化**

完成知识发现后，生成 `.arch-guardian/arch.md`，作为项目架构的单一权威文档，内容包括：

1. **架构图** — 用 ASCII 字符绘制项目整体架构（模块/分层/主要数据流），示例：

   ```
   ┌─────────────────────────────────────┐
   │              UI Layer               │
   └──────────────┬──────────────────────┘
                  │
   ┌──────────────▼──────────────────────┐
   │           Service Layer             │
   └──────────────┬──────────────────────┘
                  │
   ┌──────────────▼──────────────────────┐
   │         Repository / Data Layer     │
   └─────────────────────────────────────┘
   ```

   根据项目实际情况绘制，优先使用 ASCII 字符；若项目有既有架构图则以其为准并更新。

2. **架构知识**：
   - 识别出的技术栈
   - 参考文档路径及其类型
   - 用户补充的未成文约定
   - 外部获取的最佳实践摘要（附来源）

3. **审查报告索引** — 初始为空，每次生成新报告后自动追加一条记录：

   ```
   | 日期 | 报告文件 | 摘要 |
   |------|----------|------|
   ```

后续审查直接读取 `arch.md`，无需重复发现。当项目技术栈或架构文档有重大变化时，用户可以要求重新执行知识发现并更新 `arch.md`。

#### 后续使用：读取已有依据

如果 `.arch-guardian/arch.md` 已存在，直接读取作为审查依据。同时快速检查：
- 引用的参考文档是否仍然存在
- 是否有新增的架构相关文档未被索引
- 如果发现变化，提醒用户是否需要更新 `arch.md`

### 第四步：确定审查范围

按优先级确定范围：

1. **用户指定** — 用户明确说了要审查什么，就审查什么
2. **变更聚焦** — 如果有明确的变更上下文（正在讨论的需求、刚完成的代码），以变更为切入点，审查其对整体架构的影响
3. **全项目扫描** — 无任何上下文时，扫描整个项目的目录结构、模块组织、依赖关系

无论哪种范围，都需要先对项目全局有基本了解（目录结构、主要模块、入口点），再聚焦到具体审查目标。

### 第五步：执行审查

对每个审查项，给出以下判定之一：
- **通过** — 符合规范，无需修改
- **警告** — 不违规但存在隐患，建议关注
- **不通过** — 违反规范，需要修改
- **跳过** — 与本次审查范围无关（仅聚焦审查模式下使用）

## 架构层审查

审查整体架构的健康度。

### A1. 分层清晰度
- 项目是否有明确的分层（参照项目架构文档中定义的层级划分）
- 各层职责是否清晰，有无职责混淆
- 层与层之间的依赖方向是否合理（避免反向依赖、循环依赖）

### A2. 目录结构
- 目录组织是否能让新人快速理解项目结构
- 有无"垃圾桶"目录（如 `utils/`、`misc/`、`managers/`、`temp/`）
- 文件放置位置是否符合其职责归属

### A3. 全局依赖
- 全局状态和全局可访问对象的数量和职责是否合理
- 有无"上帝对象"承担了过多职责
- 跨模块通信方式是否统一且可追踪

### A4. 变更影响（设计审查重点关注此项）
- 新增的系统/模块是否在已有分层中有明确归属
- 是否引入了新的架构概念或模式（需要额外审慎）
- 是否打破了已有的架构约定

---

## 模块层审查

审查各模块的内聚性和模块间的耦合度。

### M1. 模块边界
- 每个模块是否有明确的职责范围
- 模块的公开接口是否最小化
- 模块内部实现是否对外隐藏

### M2. 模块间耦合
- 模块之间是否通过明确的接口或约定的通信机制交互
- 有无模块直接操作另一个模块的内部状态
- 修改一个模块是否会波及到不相关的模块

### M3. 可替换性
- 模块是否可以独立理解和测试
- 是否可以替换某个模块的实现而不影响其他部分

### M4. 数据流向
- 数据在模块间的流动路径是否清晰
- 有无数据在多个模块间被不必要地传递或复制
- 配置/数据是否与行为适当分离

---

## 组件层审查

审查具体的代码单元（文件、类、函数）。设计审查模式下跳过此层。

### C1. 单一职责
- 每个文件/类是否只做一件事
- 有无"上帝对象"承担了过多不相关的职责
- 函数长度和复杂度是否合理

### C2. 命名和可读性
- 命名是否能表达语义（而非 `Manager`、`Helper`、`Util` 等模糊名字）
- 代码组织结构是否能直观反映职责

### C3. 依赖方向
- 组件是否只依赖它应该知道的东西
- 有无越级引用（如底层组件直接操作上层对象）
- 依赖注入是否优先于硬编码引用

### C4. 可扩展性
- 新增同类对象的成本是否足够低
- 是否需要大量复制已有代码才能扩展
- 数据驱动的部分是否已适当外置

## 审查报告持久化

审查报告保存到项目中，作为架构演进的记录。

### 存储位置

- **报告目录**: `.arch-guardian/reports/`
- 用户指定了其他目录则使用用户指定的
- 首次使用时自动创建目录

### 文件命名

```
.arch-guardian/
  arch.md
  reports/
    YYYY-MM-DD-<主题摘要>.md
```

例如：
- `.arch-guardian/reports/2026-03-18-新增存档系统.md`
- `.arch-guardian/reports/2026-03-20-全项目架构扫描.md`

### 报告索引同步

每次保存报告后，在 `.arch-guardian/arch.md` 的"审查报告索引"表格中追加一行：

```
| YYYY-MM-DD | [主题摘要](reports/YYYY-MM-DD-<主题摘要>.md) | 一句话摘要 |
```

### 历史关联

- 新报告应检查是否有相关的历史报告，如有则在报告开头引用，说明架构演进轨迹
- 如果之前的"不通过"项已修复，在旧报告末尾追加更新记录：

```
## 更新记录
- **YYYY-MM-DD**: [更新说明，如"A2 目录结构问题已修复"]
```

## 架构文档资产化

架构知识不应只存在于代码中或审查者的脑中，应当沉淀为可追溯的文档资产。

### 触发时机

审查过程中如果发现以下情况，主动提醒用户：

1. **项目没有架构文档** — 建议用户创建基础架构文档，至少包含：分层定义、目录结构约定、模块职责划分、跨模块通信方式
2. **架构文档与代码不一致** — 代码实际结构已偏离文档描述，提醒用户更新文档
3. **审查中产生了新的架构决策** — 建议用户记录为 ADR（Architecture Decision Record）

### 架构文档同步检查

每次审查时，如果项目有架构文档，额外执行一次同步检查：

- 文档中描述的分层/模块是否与代码实际结构一致
- 文档中的约定是否仍在被遵守
- 是否有新增的模块/系统未被文档覆盖

同步检查结果附在审查报告末尾：

```
## 文档同步状态
- **架构文档**: [路径] — 同步 / 需更新（说明偏差）
- **ADR**: [是否存在] — 最新记录日期
```

## 变更影响速检

一种轻量级的架构检查模式，设计为在 git commit 前自动触发。它不是完整的架构审查，而是一个快速的"架构雷达"——在代码提交前扫描变更是否触及了架构敏感区域，几秒内给出结论。

### 触发方式

通过 Claude Code 的 PreToolUse hook 自动触发。当 Claude 即将执行 `git commit` 命令时，hook 脚本拦截并通过 `additionalContext` 输出显式指令要求调用本技能的速检模式。你会在 hook 输出中看到"请调用 arch-guardian 技能，执行「变更影响速检」模式"这样的指令。

收到此指令后，直接按下方"速检流程"执行，不需要额外确认。

### 速检流程

收到 hook 输出后，执行以下快速分析（不需要用户指令，自动执行）：

**1. 收集变更上下文**

用 `git diff --cached --name-only` 获取暂存区中即将提交的文件列表，这就是本次提交的完整变更集。

**2. 读取架构知识**

如果 `.arch-guardian/arch.md` 存在，快速读取其中的架构图和分层定义。如果不存在，基于目录结构做合理推断。

**3. 分析变更影响**

针对当前变更的文件，快速判断以下四个维度：

| 维度 | 检查内容 | 风险信号 |
|------|----------|----------|
| **层级归属** | 变更文件属于哪个架构层级 | 文件放在了不属于它的层级中 |
| **跨层依赖** | 变更是否引入了跨层级的新依赖 | 底层模块引用了上层模块 |
| **模块边界** | 变更是否跨越了模块边界 | 一次变更同时修改了多个不相关模块的内部 |
| **接口变更** | 变更是否修改了模块的公开接口 | 接口签名变化可能影响所有调用方 |

**4. 输出结论**

根据分析结果，输出一个简短的内联摘要（不生成报告文件）：

- **无风险** — 变更在预期范围内，不影响架构。静默通过，不输出任何内容，避免打断工作流。
- **需关注** — 检测到潜在的架构影响，输出一行简短提示。例如：

  > `[arch-guardian] 注意: services/order_service.py 新增了对 db/connection.py 的直接依赖，跳过了 models 层。建议确认这是否符合分层约定。`

- **建议审查** — 检测到较严重的架构风险，建议用户执行完整的架构审查。例如：

  > `[arch-guardian] 警告: 本次变更跨越了 3 个模块边界且修改了 2 个公开接口，建议执行完整架构审查（/arch-guardian）。`

### 速检原则

- **不打断** — 大多数变更应该静默通过并继续执行 commit。只有真正需要关注的情况才输出提示。
- **不阻塞** — 速检完成后继续执行原来的 git commit，不阻止提交。如果发现严重问题，提示用户但仍然允许提交。
- **不重复** — 如果同一问题已经提示过，不再重复提示。
- **有依据** — 每条提示都要说明具体原因，而不是笼统的"可能有问题"。

---

## 审查原则

1. **有据可依** — 每个判定都要能追溯到参考文档中的具体条款。如果是基于通用原则，也要明确说明是哪条原则。
2. **就事论事** — 只审查架构层面的问题，不纠结代码风格细节（那是 code review 的事）。组件层的审查也聚焦在架构意义上的问题（职责划分、依赖方向），而非语法偏好。
3. **务实判断** — 不追求理论完美。一个"足够好"的架构优于一个"理论完美但过度设计"的架构。如果某处不够理想但当前规模下完全可控，给"警告"而非"不通过"。
4. **考虑阶段** — 原型阶段和生产阶段的标准不同。对早期项目宽容一些，对成熟项目严格一些。
5. **建议可行** — 给出的建议必须是用户当前就能采取行动的，而不是"以后重写整个项目"。
